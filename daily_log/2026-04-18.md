## In progress / uncommitted work

The submain working tree has significant uncommitted changes across five modified files and 21 untracked files. This is the audit Part 2 sprint, fixing three of the four failures identified in the Part 1 audit.

### Recursive type parser (`src/frontend/parser.rs`, +67/-61)

The `type_parser` function has been refactored from a flat combinator chain to a `recursive()` closure. This unlocks:
- `Result<Result<t32>>` — nested Result types that previously failed to parse (audit_02 fix)
- `Handle<T>` — now parseable as a first-class type
- `[N: Result<T>]` — arrays of complex types

The old parser used `scalar.clone().or(named_type.clone())` for the inner type of `Result<T>`, which couldn't match the `Result` keyword token recursively. The new parser passes the recursive `ty` reference into all sub-parsers.

### Struct field type resolution in semantic pass (`src/frontend/semantic.rs`, +33/-2)

Two code paths updated:

1. **Assignment to struct fields** — the semantic pass now looks up the actual field type from the struct definition for `DotAccess` LValues. It resolves the struct name from the instance variable, finds the field in `self.structs`, and applies `types_compatible` checking with a type mismatch error. Also applies `insert_cast_if_needed` for numeric coercion. This fixes audit_09 at the semantic level.

2. **Compound assignment** — same field type lookup applied to `CompoundAssign` targets on struct fields. The `ty` field on `SemanticLValue::DotAccess` now carries the real field type instead of `SemanticType::Unknown`.

### 64 MB interpreter thread (`src/main.rs`, +27/-1)

The interpreter now runs on a dedicated thread with a 64 MB stack via `std::thread::Builder::new().stack_size(64 * 1024 * 1024)`. The old code ran directly in `main()` on the default 1 MB Windows thread stack. `fib(10)` generated ~177 recursive Cx-level calls, each burning multiple KB of Rust stack, causing a stack overflow.

The fix is pragmatic — a comment notes that reducing per-frame stack consumption is a post-0.1 optimization. The 64 MB headroom handles any reasonable recursive Cx code. `fib(15)` now produces 610 correctly (t113 matrix test). This fixes audit_11.

### Struct field truncation in runtime (`src/runtime/runtime.rs`, +20/-8)

Both `Assign` and `CompoundAssign` arms now call `apply_numeric_cast(val, ty)` using the LValue's type field before storing values. This applies to both `Binding` and `DotAccess` variants. Previously, compound assignment on struct fields (`c.value += 10` where `value: t8`) bypassed width truncation entirely, producing full i128 results. This fixes audit_09 at the runtime level.

## Decisions and direction

The audit Part 2 sprint confirms three concrete technical decisions:

1. **Recursive type composition is supported.** The type parser now uses Chumsky's `recursive()` combinator, meaning `Result<Result<T>>`, `Handle<Result<T>>`, `[N: Handle<T>]` and similar compositions work structurally. This is a quiet but important parser architecture change.

2. **Width truncation is enforcement-complete for struct fields.** The overflow enforcement sprint (committed as `dc564a7`) handled arithmetic expressions. The uncommitted work closes the gap for struct field writes — both direct assignment and compound assignment now apply `apply_numeric_cast` through the LValue's type field.

3. **Deep recursion solved pragmatically.** Rather than rewriting the interpreter to use an explicit call stack (a major architectural change), a 64 MB thread provides generous headroom. The per-frame stack consumption is documented as a post-0.1 optimization.
