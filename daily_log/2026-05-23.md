## Landed today

**refactor(frontend): add Type::Void AST variant (f3c2856, merged to main via PR #278).** CONFIRMED. Previously, `void` in function signatures was parsed as an identifier and resolved through the struct-type path. This commit gives it first-class representation: `Token::TypeVoid` in the lexer, `Type::Void` in the AST enum, direct mapping in the parser, and `SemanticType::Void` routing in the semantic phase. Two new unit tests verify the round trip — one at the parser level confirming `fnc: void main() {}` produces `Type::Void`, one at the semantic level confirming that a void-returning function with a trailing void call (like `print()`) passes analysis. The merge commit is `2a2e1b2`.

**feat(runtime): add exit() builtin for explicit exit code propagation (4d612df, on submain).** CONFIRMED. This is a substantial commit touching 25 files across four layers of the compiler:

- *Types:* Added `RuntimeError::Exit(i32)` as a control-flow signal in `types.rs`, sibling to `EarlyReturn` and `BreakSignal`.
- *Semantic:* Added `exit()` analysis in `semantic.rs` — accepts 0 or 1 argument, arity enforced at analysis time, returns `SemanticType::Void`.
- *Runtime:* Implemented the builtin in `runtime.rs` — evaluates the optional integer argument, validates it fits i32, returns the `Exit` signal. Out-of-range values surface as `TypeMismatch`.
- *Main loop:* Two catch sites in `main.rs`. In normal interpreter mode, the `Exit` signal is caught before error rendering, stdout is flushed (to prevent lost piped output since `process::exit` skips Drop), and `process::exit(code)` is called. In `--test` mode, `exit(0)` counts as PASS and `exit(n!=0)` counts as FAIL, with the test runner continuing to the next test — this prevents `exit()` from silently disabling the rest of the suite.
- *Diagnostics:* Added a never-render arm in `diagnostics.rs` for `RuntimeError::Exit` as a safety net.
- *Matrix harness:* Extended `run_matrix.sh` with `.expected_exit` file support, allowing fixtures to assert specific exit codes and optionally verify stdout.
- *Test fixtures:* Seven new verification matrix tests: `t_exit_default` (no-arg defaults to 0), `t_exit_zero_explicit`, `t_exit_nonzero` (exit 3), `t_exit_before_output` (exit before any print), `t_exit_after_partial` (exit after partial output, verifies stdout flush), `t_exit_in_branch` (exit from if-branch), `t_exit_in_function` (exit propagates past function-call boundary).
- *Integration tests:* Two new tests in `diff_harness.rs` verify the --test mode behavior: `exit(1)` in a `#[test]` function records as FAIL and the runner continues, `exit(0)` records as PASS and the runner continues.

This commit is branch-local to submain and not yet on main.

## Decisions and direction

**`void` as a first-class type.** The Type::Void refactor is a deliberate language-model decision: `void` is now a keyword with its own token and AST variant, not an identifier that resolves through the struct/type path. This eliminates a class of edge-case bugs where `void` could shadow or collide with user-defined types, and it makes the parser's type resolution path explicit. It's the kind of post-release cleanup that firms up the language's type system representation.

**exit() as a control-flow signal, not a process kill.** The design choice to make `exit()` propagate as a `RuntimeError::Exit(i32)` signal rather than calling `process::exit` directly at the call site is significant. It means exit() composes correctly with the test runner (where it becomes a test result rather than a process terminator), it allows stdout to be flushed before termination, and it keeps the runtime's control-flow model consistent — Exit is a sibling of EarlyReturn, BreakSignal, and ContinueSignal, all carried as RuntimeError variants. The --test mode integration is especially thoughtful: without it, a `#[test]` function calling `exit(1)` would kill the entire test suite silently.
