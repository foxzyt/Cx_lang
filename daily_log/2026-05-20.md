## Decisions and direction

**Method-call lowering chose synthesis over duplication.** The MethodCall arm at `lower.rs:1791` could have replicated the Call lowering logic for method dispatch. Instead, 0ab7e9b synthesizes a Call expression (receiver as first arg) and recurses into the existing Call arm. This keeps method dispatch as a thin translation layer rather than a parallel code path — consistent with the project's existing pattern of reducing surface area through delegation.

**Literal-width narrowing completed as a family.** The method-call work exposed gaps in how `Numeric` literals interact with typed peers across the operator surface. Rather than fixing only MethodCall args, 0ab7e9b closed the entire gap: assert_eq peer-narrowing, Lt/Gt/LtEq/GtEq operators, and MethodCall args all now use `insert_cast_if_needed`. This is the kind of "fix the class, not the instance" decision that prevents recurring one-off patches.

**Print widening takes a per-type-family approach.** Rather than a single widening intrinsic, 08fa2f9 routes by type family: integers sign-extend to I64 via Cast and call `cx_printn`, Bools call a separate `cx_print_bool` intrinsic. F64 and I128 are explicitly parked post-0.1. This avoids a complex polymorphic print path while unblocking the 15 matrix fixtures that were print-stub-blocked.

**0.1 readiness audit formalized as inline annotations.** The two roadmap audit commits (1036fd8, f9fcae5) embed per-item verification directly in the roadmap file as HTML comments. This makes the audit results co-located with the claims they verify — reviewers can see `<!-- audit: VERIFIED -->` or `<!-- audit: DRIFTED -->` next to each checkbox. The audit identified that the Cast category appearing green in the parity matrix is misleading — the PASSes came from narrowing ride-alongs, not from the actual CX-91 Cast JIT work.
