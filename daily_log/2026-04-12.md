## Landed today

**CONFIRMED — `bcd49f2` on `origin/submain`: Phase 10 loop / break / continue lowering** (`src/ir/lower.rs`, +215/−16)
Threads `LoopContext` (header_id, exit_id, ordered_bindings) through statement lowering so that structured jumps know where to go. Concretely:
- `lower_loop` added: infinite-loop CFG. Entry → header (unconditional jump into body) → body; exit block created with block params matching the loop-carried binding set.
- `lower_while` updated: `exit_block` is now constructed with proper block params (one per ordered binding) and the header's `Branch` terminator passes real `else_args` into it instead of an empty vector. This is the bug fix that makes loop-carried SSA actually survive the while's early-exit edge.
- `SemanticStmt::Break` lowered: gather current SSA values for every ordered binding, emit `IrTerminator::Jump { target: exit_id, args: exit_args }`, seal.
- `SemanticStmt::Continue` lowered: same pattern, target `header_id` instead.
- `loop_ctx: Option<&LoopContext>` is plumbed through `lower_stmt`, `lower_if_else`, and `lower_if_chain` so a `break` inside a nested `if` inside a loop resolves to the enclosing loop's exit. Previously `lower_if_chain` hardcoded `None` for its inner sequence — that's now `loop_ctx`.
- Error paths: `break`/`continue` outside a loop return `UnsupportedSemanticConstruct`; missing bindings in the SSA env return `InternalInvariantViolation`.
- The three existing "rejects_unsupported_statement" tests were retargeted from `SemanticStmt::Loop` to `SemanticStmt::For` (which is the next unsupported stmt on the list). `SemanticStmt::For` is still rejected — only `Loop`, `Break`, `Continue` flipped from rejected to lowered today.
- Fallout additions: `SemanticExprKind::ResultOk`, `ResultErr`, `Try`, and `SemanticType::Result(_)` each got an `unsupported!` arm so the match stays exhaustive — this hints the semantic layer has already grown Result shapes on submain, but lowering does not implement them yet.
